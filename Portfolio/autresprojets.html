SITE RESPONSIVE

Un site pensé pour tous les écrans, c'est plus qu'un simple confort : c'est une nécessité

Un site pensé pour tous les écrans, c'est plus qu'un simple confort : c'est une nécessité

Aujourd’hui, les utilisateurs naviguent autant sur téléphone, tablette que sur ordinateur. Un site responsive est conçu pour s’ajuster automatiquement à tous ces formats.

Derrière cette souplesse, il y a un vrai travail technique : on utilise des grilles flexibles, des points de rupture (breakpoints) selon les tailles d’écran, et du CSS adaptatif pour faire évoluer la mise en page en temps réel.

Ce projet m’a permis de concevoir une interface claire, cohérente et facile à utiliser, aussi bien sur un petit écran que sur un grand. L’objectif : allier esthétique, confort de lecture et performance, sans compromis.

SÉCURISATION D'UN SITE AVEC OWASP

Apprendre à protéger une application web dès ses premières lignes de code, c’était l’objectif de ce projet de sécurisation en PHP, réalisé selon les principes de l’OWASP.

Apprendre à protéger une application web dès ses premières lignes de code, c’était l’objectif de ce projet de sécurisation en PHP, réalisé selon les principes de l’OWASP.





Pour les moins expérimentés : il s’agit ici d’une requête préparée, une technique qui permet de se prémunir contre les injections SQL. Tout simplement l’une des failles les plus connues et exploitées sur le web.

L’idée est simple : au lieu qu’un utilisateur mal intentionné puisse renseigner un bout de code dans un formulaire, ce qui viendrait tromper le système et pourrait entraîner de lourdes conséquences, ici on fait en sorte que le système ne puisse jamais interpréter ce que l’utilisateur saisit comme du code.

Les plus expérimentés auront remarqué l’usage de SHA-256 pour le hachage du mot de passe. Cette méthode, bien que couramment rencontrée dans des tutoriels anciens, est aujourd’hui à éviter dans un environnement réel.

Ce choix a été fait ici dans un cadre pédagogique : l'objectif était avant tout de comprendre le fonctionnement d'une requête sécurisée, et non d'implémenter un système d'authentification complet.

En approfondissant mes connaissances, j’ai découvert qu’il est préférable d’utiliser les fonctions natives de PHP comme password_hash() et password_verify(), qui sont plus sûres car elles incluent automatiquement un sel cryptographique et utilisent des algorithmes conçus pour le hachage de mots de passe.

Cette première approche de la sécurité web m’a permis de comprendre l’importance d’isoler les données utilisateurs et d’appliquer des bonnes pratiques dès le départ pour construire des applications robustes.


TESTS UNITAIRES EN JAVA

Dans ce projet, l’objectif était de mettre en place des tests unitaires avec JUnit, afin de valider le bon fonctionnement d’une calculatrice codée en Java.

Dans ce projet, l’objectif était de mettre en place des tests unitaires avec JUnit, afin de valider le bon fonctionnement d’une calculatrice codée en Java.

Chaque opération (addition, soustraction, multiplication, division) a été testée avec différents cas : valeurs positives, négatives, et même la division par zéro, pour s’assurer de la fiabilité de l’algorithme dans toutes les situations.





Ce que montre l’extrait ci-dessus : la structure typique d’un test unitaire. On commence par préparer l’environnement, on appelle la méthode à tester, puis on vérifie le résultat attendu grâce à la fonction assertEquals().

Ces tests permettent de détecter automatiquement toute anomalie, par exemple si une méthode retourne un résultat incorrect après une modification. C’est une bonne pratique essentielle dans les projets de développement professionnels.


SERVEUR DNS

Ce projet avait pour objectif de déployer un environnement serveur complet via Windows Server 2016 dans une machine virtuelle, en installant les rôles
AD DS (Active Directory Domain Services) et DNS.

Ce projet avait pour objectif de déployer un environnement serveur complet via Windows Server 2016 dans une machine virtuelle, en installant les rôles
AD DS (Active Directory Domain Services) et DNS.

Windows Server 2016 a été choisi pour sa stabilité et sa compatibilité avec les fonctionnalités avancées comme Active Directory. Une fois le rôle AD installé, le serveur DNS est automatiquement configuré, car il est nécessaire pour la gestion des noms dans un domaine.





Le tableau de bord confirme que les rôles ont été correctement installés et sont opérationnels.







Un test DNS a été réalisé à l’aide de la commande nslookup, qui permet de vérifier la résolution des noms de domaine. Le serveur répond bien, preuve que le DNS est fonctionnel.





Enfin, la commande netdom query fsmo montre que le serveur occupe tous les rôles FSMO, ce qui confirme qu’il est bien contrôleur de domaine principal.



Ce projet m’a permis de mieux comprendre les bases du réseau d’entreprise, la gestion des domaines via Active Directory, et le fonctionnement interne du DNS dans un environnement Windows Server.

Même s’il ne s’agit pas d’une configuration d’entreprise complète, cette première mise en place m’a permis de me familiariser avec les outils indispensables en administration système.

GLPI est un outil open-source conçu pour gérer les infrastructures informatiques et le parc matériel d'une organisation. GPO

Une GPO (Group Policy Object) est un outil essentiel permettant aux administrateurs système de déployer et appliquer des stratégies sur les ordinateurs et les utilisateurs d’un domaine Active Directory.

Une GPO (Group Policy Object) est un outil essentiel permettant aux administrateurs système de déployer et appliquer des stratégies sur les ordinateurs et les utilisateurs d’un domaine Active Directory.

Elle facilite l'automatisation de configurations et l'application de règles sur l’ensemble du réseau.

Les GPO permettent de gérer divers aspects des postes de travail, comme les paramètres réseau, les restrictions d'accès, et l'exécution automatique de scripts. Elles sont couramment utilisées pour améliorer la sécurité et optimiser la gestion informatique d'une organisation.

Dans mon projet, j’ai mis en place une GPO visant à afficher une image de présentation sur une page web lors de l’ouverture des sessions utilisateurs. Voici les principales étapes de son déploiement :

  • Création d'une nouvelle GPO via la console de gestion des stratégies de groupe.
  • Ajout d’un script PowerShell directement dans la GPO pour ouvrir l’image de présentation dans un navigateur.
  • Définition de la GPO pour exécuter le script à l’ouverture de la session utilisateur.
  • Application de la GPO aux utilisateurs concernés et vérification du bon fonctionnement.

Grâce à cette GPO, l’affichage personnalisé au démarrage des sessions est centralisé, garantissant une meilleure cohérence visuelle et informative pour les utilisateurs.
